# Architecture Decision Records (ADR)

의사결정의 **과정**을 기록한다. 결론은 코드에 있다. ADR에는 왜 다른 선택지를 버렸는지를 적는다.

## 파일 네이밍

```
ALGOGO-{이슈번호}-{작업요약}.md
```

## 작성 기준

**다른 방법으로 해도 됐는데 이걸 골랐으면 쓴다. 이렇게 해야 하는 게 정해져 있었으면 안 쓴다.**

커밋 타입(feat/refactor/fix)과 무관하다. 작업 내용이 다음에 해당하는지로 판단한다:

### 작성 O

| 상황 | 판단 근거 |
|------|-----------|
| 같은 기능을 구현하는 방법이 여러 개 있었다 | 패턴, 라이브러리, 알고리즘 등에서 선택지가 있었다 |
| 코드를 어느 모듈/레이어에 둘지 고민했다 | 위치에 따라 의존성 구조가 달라진다 |
| 무언가를 포기하고 다른 것을 얻었다 | 성능 vs 가독성, 단순함 vs 확장성 등 트레이드오프 |
| 일반적 관행과 다르게 했다 | 프로젝트 맥락에서 관행이 맞지 않는 이유가 있었다 |

### 작성 X

| 상황 | 판단 근거 |
|------|-----------|
| 코딩 규칙에 이미 정의되어 있다 | 규칙대로 따랐을 뿐, 판단이 없었다 |
| 고쳐야 할 코드가 명확하다 | 버그 수정, 누락된 await, 잘못된 타입 등 |
| 기계적으로 반복 적용했다 | import 경로 통일, 상수 추출, 포맷팅 등 |
| 업계 표준/베스트 프랙티스를 그대로 적용했다 | 선택이 아니라 정답이었다 |

## 핵심: 과정을 기록한다

ADR의 가치는 결론이 아니라 **어떻게 그 결론에 도달했는가**이다.

- **어떤 문제가 있었는지** — 현재 상태의 어떤 점이 불편하거나 잘못되었는지
- **무엇을 검토했는지** — 어떤 선택지들을 비교했는지, 각각의 장단점
- **왜 버렸는지** — 선택하지 않은 대안을 버린 구체적 이유
- **왜 골랐는지** — 선택한 대안이 이 프로젝트에서 나은 이유

"A를 선택했다"는 PR에서 알 수 있다. "B와 C를 검토했지만 X 때문에 버렸다"는 ADR에서만 알 수 있다.

## 템플릿

```markdown
# ALGOGO-{번호}: {작업 제목}

## 문제

무엇이 문제였는지. 현재 상태의 어떤 점이 불편하거나 잘못되었는지.

## 검토한 대안

| 대안 | 설명 | 장점 | 단점 |
|------|------|------|------|
| A | ... | ... | ... |
| B | ... | ... | ... |

## 결정

어떤 대안을 선택했는지, 한 문장으로.

## 이유

왜 이 대안을 선택했는지. 다른 대안을 버린 이유도 포함.

## 참고

- 관련 이슈, PR, 외부 자료 링크
```

## 규칙

- **인과관계를 명확히**: "X라는 문제가 있었다 → Y를 검토했다 → Z를 선택했다 → 이유는 W" 흐름을 유지한다
- **결과가 아닌 과정을 기록**: 코드가 어떻게 바뀌었는지는 PR에 있다. ADR에는 왜 그렇게 바꿨는지를 적는다
- **짧게 쓴다**: 한 문서당 읽는 데 2분 이내. 장황하면 핵심이 묻힌다
- **사후에도 작성 가능**: 이미 완료된 작업이라도 의사결정 과정이 중요하면 소급 작성한다
