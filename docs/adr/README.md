# Architecture Decision Records (ADR)

작업별 의사결정 과정을 기록한다. 사람과 AI 모두가 읽고, 왜 이런 결정을 했는지 추적할 수 있어야 한다.

## 파일 네이밍

```
ALGOGO-{이슈번호}-{작업요약}.md
```

예시:
- `ALGOGO-1-auth-token-storage.md`
- `ALGOGO-5-websocket-execution-pipeline.md`

## 작성 기준

**대안을 검토하고 하나를 선택한 의사결정이 있었으면 작성한다.** 커밋 타입(feat/refactor/fix)과 무관하다.

구체적으로:
1. **관행과 다른 선택** — 일반적인 방식 대신 프로젝트에 맞는 다른 방식을 택한 경우
2. **대안이 있었는데 하나를 고른 경우** — 라이브러리, 패턴, 구조, 정책 등에서 여러 옵션을 비교한 경우
3. **제한/트레이드오프가 있는 결정** — 성능, 메모리, 호환성, 사용성 등에서 절충한 경우

선택의 여지가 없는 작업(버그 수정, 규칙에 정해진 대로 따른 변경, 기계적 정리)은 제외.

## 템플릿

```markdown
# ALGOGO-{번호}: {작업 제목}

## 문제

무엇이 문제였는지. 현재 상태의 어떤 점이 불편하거나 잘못되었는지 서술한다.

## 검토한 대안

| 대안 | 설명 | 장점 | 단점 |
|------|------|------|------|
| A | ... | ... | ... |
| B | ... | ... | ... |

## 결정

어떤 대안을 선택했는지, 한 문장으로.

## 이유

왜 이 대안을 선택했는지. 다른 대안을 버린 이유도 포함.

## 참고

- 관련 이슈, PR, 외부 자료 링크
```

## 규칙

- **인과관계를 명확히**: "X라는 문제가 있었다 → Y를 검토했다 → Z를 선택했다 → 이유는 W" 흐름을 유지한다
- **결과가 아닌 과정을 기록**: 코드가 어떻게 바뀌었는지는 PR에 있다. ADR에는 왜 그렇게 바꿨는지를 적는다
- **짧게 쓴다**: 한 문서당 읽는 데 2분 이내. 장황하면 핵심이 묻힌다
- **사후에도 작성 가능**: 이미 완료된 작업이라도 의사결정 과정이 중요하면 소급 작성한다
