# Git 규칙

## 브랜치 전략

### 기본 브랜치

| 브랜치 | 용도 |
|--------|------|
| `main` | 릴리스 가능한 안정 버전 |
| `dev` | 개발 통합 브랜치. 모든 기능 브랜치는 여기서 분기하고 여기로 머지 |

### 기능 브랜치

Linear 이슈 티켓명을 그대로 브랜치명으로 사용한다.

```
ALGOGO-{번호}
```

예시:
```
ALGOGO-5
ALGOGO-12
ALGOGO-25
```

### 브랜치 생성 규칙

- 항상 `dev`에서 분기한다.
- 예외: 긴급 수정은 `main`에서 분기하고, `main`과 `dev` 양쪽에 머지한다.
- 하나의 브랜치는 하나의 이슈에 대응한다.
- Sub-issue가 있는 경우, 부모 이슈 브랜치 하나에서 작업한다.

## 커밋 컨벤션

### 형식

```
[ALGOGO-{번호}] {type}: {제목}

{본문 (선택)}
```

- 모든 커밋에 Linear 이슈 번호를 포함한다. 이슈 번호 없는 커밋은 허용하지 않는다.
- 이슈 번호는 현재 브랜치명에서 `ALGOGO-\d+` 패턴을 추출하여 사용한다.
- 제목은 한글로 작성한다. type은 영문, 기술 용어도 영문 그대로 사용 가능.
- 마침표를 붙이지 않는다.

**type 종류:**

| type | 사용 시점 |
|------|----------|
| `feat` | 새 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 동작 변경 없이 코드 구조 개선 |
| `chore` | 빌드, 설정, 의존성 등 |
| `docs` | 문서 추가/수정 |
| `test` | 테스트 추가/수정 |
| `style` | 코드 포맷, 공백, 세미콜론 등 (동작 변경 없음) |

> hotfix 브랜치에서도 커밋 type은 `fix`를 사용한다.

### 커밋 단위

- 하나의 커밋은 하나의 논리적 변경에 대응한다.
- 여러 목적의 변경을 하나의 커밋에 섞지 않는다.
- 커밋 단위는 되돌렸을 때 의미 있는 단위여야 한다.

```
좋은 예:
  커밋 1: [ALGOGO-5] feat: OAuth 로그인 서비스 구현
  커밋 2: [ALGOGO-5] feat: refresh token Redis 저장 로직 추가
  커밋 3: [ALGOGO-5] test: OAuth 로그인 테스트 작성

나쁜 예:
  커밋 1: [ALGOGO-5] feat: OAuth 구현 + 설정 파일 수정 + 테스트 추가
```

### 본문

- 제목과 빈 줄로 구분한다.
- 변경의 이유(why)가 제목만으로 전달되지 않을 때 작성한다.
- 단순한 커밋은 본문을 생략할 수 있다.

### 예시

```
[ALGOGO-5] feat: 문제 실행 WebSocket 게이트웨이 구현

BullMQ를 통한 비동기 코드 실행과 Socket.IO 실시간 결과 전달 포함.
```

```
[ALGOGO-12] fix: Redis refresh token 만료 미설정 오류 수정

TTL 없이 저장되어 만료되지 않는 토큰이 누적되고 있었다.
```

```
[ALGOGO-3] docs: API 문서 업데이트
```

## PR 규칙

### 생성 시점

기능 브랜치에서 작업이 완료된 후 PR을 생성한다.

### 제목

```
[ALGOGO-{번호}] {type}: {설명}
```

커밋과 동일한 형식. PR이 여러 커밋을 포함하면 전체 작업을 요약한다. 이슈 번호는 브랜치명에서 파싱한다.

### 본문

```markdown
## 작업 내용
- [ ] 수행한 작업 1
- [ ] 수행한 작업 2
- [ ] 수행한 작업 3

## 테스트
- [ ] 수행한 테스트 1
- [ ] 수행한 테스트 2

## 관련 이슈
- ALGOGO-{번호}
```

- **작업 내용**: 이 PR에서 실제로 한 작업을 체크리스트로 기록한다. PR 생성 시 완료된 항목은 반드시 `[x]`로 표시한다.
- **테스트**: 어떤 테스트를 수행했는지 체크리스트로 기록한다. 문서만 수정한 경우 등 테스트가 불필요하면 `해당 없음: {사유}` 로 적는다.
- **관련 이슈**: Linear 이슈 번호를 적는다.

### 머지 규칙

| 방향 | 머지 방식 | 커밋 메시지 |
|------|-----------|------------|
| 기능 브랜치 → `dev` | squash merge | PR 제목을 사용한다 |
| `dev` → `main` | **PR merge commit** | 기본 머지 메시지 사용 |
| `hotfix` → `main` | PR merge commit | 기본 머지 메시지 사용 |
| `hotfix` → `dev` | merge commit | 기본 머지 메시지를 사용한다 |

- 셀프 리뷰 후 머지를 허용한다.
- 머지 후 기능 브랜치는 삭제한다.
- 기능 브랜치에서 충돌 발생 시 `dev`를 기능 브랜치에 merge하여 해결한다 (rebase 하지 않는다).

### dev → main 머지 시점

- **릴리스 시에만** dev를 main에 머지한다 (Phase 완료, 버전 태그 등).
- 수시로 머지하지 않는다. dev에서 작업하다가 릴리스할 때만 올린다.
- main에 직접 커밋하지 않는다. 모든 변경은 dev를 거친다 (hotfix 제외).
- main과 dev가 diverge하면 안 된다. merge commit을 사용하여 공유 히스토리를 유지한다.
- 로컬에서 main에 직접 머지하지 않는다. 반드시 GitHub PR을 통해 머지한다.
- 릴리스 PR 본문에는 포함된 모든 이슈와 변경 사항을 요약한다.

### 브랜치 보호

- **main**: 직접 push 불가, PR을 통해서만 머지.
- **dev**: 직접 push 불가, 기능 브랜치에서 PR squash merge로만 머지.
