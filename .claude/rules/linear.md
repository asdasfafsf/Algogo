## 계층 구조

```
Project (기능 단위 묶음, 기한 있음)
  └── Issue (실제 작업 단위)
        └── Sub-issue (이슈를 더 쪼갠 것)
```

### Project

구체적인 기능 묶음. 기한이 있고 완료 조건이 명확하다.

### Issue

실제 작업 단위. 한 사람이 맡아서 완료할 수 있는 크기.

### Sub-issue

이슈가 클 때 하위로 쪼갠 것. 추정치 8 이상이면 반드시 분할한다.

부모 이슈는 모든 Sub-issue가 Done이 되면 Done으로 전환한다.

## 이슈 작성 규칙

### 필드 구분

| 필드 | 구분 | 미지정 시 기본 동작 |
|------|------|-------------------|
| 제목 | **필수** | - |
| 라벨 | **필수** | 유형이 불분명하면 사용자에게 확인 |
| 설명 | **권장** | 추정치 1~2이고 제목만으로 충분하면 생략 가능 |
| 우선순위 | 선택 | No Priority로 생성. 사이클에 넣을 때 지정 |
| 추정치 | 선택 | 미지정으로 생성. 작업 시작 전 사용자와 합의 |
| Project | **필수** | 현재 프로젝트(Algogo-nest)에 배정 |
| 담당자 | 선택 | 미지정 |

### 제목

한글로 간결하게. 무엇을 하는지 한눈에 보여야 한다.

```
좋은 예:
  로그인 페이지 비밀번호 유효성 검사 추가
  대시보드 차트 렌더링 오류 수정
  문제 목록 페이지네이션 구현

나쁜 예:
  로그인 관련 작업
  기능 추가
  버그 수정
```

### 설명 (유형별 템플릿)

#### Feature (새 기능)

```markdown
## 배경
왜 이 기능이 필요한지.

## 관련
- 상위 이슈: (있으면 링크)
- 설계 문서: (있으면 링크)

## 할 일
- [ ] 구체적인 작업 항목 1
- [ ] 구체적인 작업 항목 2

## 완료 조건
- 이 이슈가 끝났다고 판단할 수 있는 객관적 기준

## 결과
- (완료 시 PR 링크를 여기에 추가)
```

#### Bug (버그)

```markdown
## 재현 경로
1. 어떤 조건에서
2. 무엇을 하면
3. 어떤 문제가 발생하는지

## 기대 동작
- 정상적으로 어떻게 동작해야 하는지

## 실제 동작
- 현재 어떻게 동작하는지

## 관련
- 발견 경위: (어떤 작업 중 발견했는지, 관련 이슈 링크)
- 원인 추정: (있으면)

## 완료 조건
- 재현 경로대로 수행했을 때 기대 동작이 나올 것

## 결과
- (완료 시 PR 링크를 여기에 추가)
```

#### Improvement (개선)

```markdown
## 현재 상태
- 지금 어떻게 동작하고, 무엇이 문제인지

## 개선 방향
- 어떻게 바꾸고 싶은지

## 관련
- 관련 이슈: (있으면 링크)

## 완료 조건
- 개선이 적용되었다고 판단할 수 있는 기준

## 결과
- (완료 시 PR 링크를 여기에 추가)
```

#### 공통 규칙

- **관련** 섹션: 이 작업이 어디서 파생됐는지, 어떤 이슈와 연결되는지를 기록한다. 독립적인 이슈면 생략 가능.
- **결과** 섹션: 이슈를 Done으로 옮길 때 PR 링크를 추가하여 작업 결과물을 추적한다.

### 라벨

| 라벨 | 사용 시점 |
|------|----------|
| Feature | 없던 기능을 새로 추가 |
| Bug | 기존에 동작하던 것이 깨진 경우 |
| Improvement | 동작은 하지만 더 나아지길 원하는 것 (리팩터링, 성능 개선, 문서 정리, 테스트 추가 등) |

하나의 이슈에 라벨은 하나만 붙인다.

**판단 기준**: 기존 기능의 동작 방식을 바꾸거나 확장하면 Improvement, 아예 없던 것을 만들면 Feature. chore, docs, test 류 작업은 Improvement로 분류한다.

### 우선순위

| 단계 | 기준 |
|------|------|
| Urgent | 다른 작업을 블로킹. 당장 해결해야 함 |
| High | 현재 프로젝트의 핵심 기능 |
| Medium | 현재 프로젝트에 포함되지만 핵심 경로는 아닌 것 |
| Low | 있으면 좋지만 미뤄도 되는 것 |
| No Priority | Backlog에 넣어둔 아이디어. 사이클에 넣을 때 반드시 우선순위를 지정한다 |

|--------|------|

## 상태 전환

```
Backlog → Todo → In Progress → In Review → Done
                       ↑            │
                       └────────────┘ (문제 발견 시)
```

| 전환 | 시점 |
|------|------|
| → Todo | 사이클에 포함되거나 작업 범위에 합의됨 |
| → In Progress | 코드 작성 시작. 생성 직후 바로 작업하면 In Progress로 바로 생성 |
| → In Review | PR 생성 |
| → Done | 리뷰 통과, 테스트 확인, 머지 완료. 결과 섹션에 PR 링크 추가 |
| → Canceled | 작업이 불필요해짐. 사유를 코멘트로 남긴다 |
| → Duplicate | 중복 이슈. 원본 이슈를 코멘트에 링크한다 |

### 할일 체크 규칙

이슈 설명의 체크리스트(`- [ ]`)는 작업 완료 시 한꺼번에 체크하지 않는다. 각 항목을 완료할 때마다 즉시 `- [x]`로 업데이트한다.

### 코멘트 규칙

다음 상황에서는 이슈에 코멘트를 남겨 맥락을 기록한다:

- 블로커를 발견했을 때
- 스펙이 변경되었을 때
- 상태를 되돌릴 때
- 오래(3일 이상) In Progress 상태일 때

## AI 에이전트 이슈 생성 플로우

### 생성 절차

```
1. 사용자 발화에서 이슈 생성 의도 감지
2. 중복 의심 시 사용자에게 확인 (기존 활성 이슈 제목과 비교)
3. 유형(라벨) 판단이 애매하면 사용자에게 확인
4. 추정치가 8 이상으로 판단되면 Sub-issue 분할을 사용자에게 제안
5. 이슈 초안(제목, 유형, 설명)을 사용자에게 보여주고 확인받음
6. LINEAR_API_KEY로 GraphQL API 호출하여 이슈 생성
7. 생성된 이슈 번호와 URL을 사용자에게 전달
```

### 필드 처리 규칙

- **제목, 라벨**: 반드시 채운다. 판단이 어려우면 사용자에게 확인.
- **우선순위**: 사용자가 언급하지 않으면 No Priority로 생성.
- **추정치**: 사용자가 언급하지 않으면 미지정으로 생성.
- **설명**: 대화 내용에서 핵심만 뽑아 해당 유형의 템플릿에 맞춰 작성.
- **Project**: 현재 프로젝트(Algogo-nest)에 배정한다.

### 상태 관리 규칙

- 사용자가 "지금 하자", "바로 시작" 등 명시적으로 즉시 작업을 요청한 경우 → In Progress로 생성.
- 그 외 → Backlog로 생성한다.
- PR 생성 시 → In Review로 전환한다. 셀프 리뷰로 바로 머지하는 경우 In Review를 건너뛰어도 된다.
- 머지 완료 시 → Done으로 전환하고 결과 섹션에 PR 링크를 추가한다.

### 금지 사항

- 사용자 동의 없이 임의로 이슈를 생성하지 않는다.
- 추정치와 우선순위를 임의로 정하지 않는다.
- 브랜치명과 커밋에 이슈 번호를 반드시 포함한다 (Git 규칙 참조).
- 이슈 없이 작업을 시작하지 않는다. 아무리 작은 작업이라도 반드시 이슈를 먼저 만들고 별도 브랜치에서 진행한다.
